# Knowledge Register

<!-- Append-only. Add concise project-specific rules, gotchas, and reusable patterns. -->

- M006/S02: If homepage verification on the worktree server shows a blank body with `__NEXT_DATA__` for `/_error`, inspect browser page source immediately; Clerk middleware can fail before any landing UI renders, making browser text assertions meaningless until auth config is fixed.
- M006/S03: If port 3001 is already occupied during worktree verification, inspect the existing background shell/server before restarting anything; the current blocker surfaces as an invalid Clerk publishable key runtime error, so browser shell assertions are not meaningful until auth config is fixed.
- M006/S03: The authenticated shell Playwright proof lives in `apps/web/tests/app-shell.spec.ts` and intentionally classifies middleware HTML containing Clerk/auth/env failures as a blocker before asserting shell selectors; use page source, not browser console output, when the render dies before hydration.
- M006/S04: In this worktree, Playwright resolves correctly from `apps/web` package context (`cd apps/web && pnpm exec playwright ...`), while the slice-plan form `pnpm --filter web-app exec playwright ...` fails with `Command "playwright" not found`; treat that as a workspace invocation issue, not a missing test dependency.
- M006/S04: Playwright’s `webServer` on port 3001 can still fail with `EADDRINUSE` even when an agent-started dev server already exists; check `lsof -iTCP:3001 -sTCP:LISTEN` before assuming app-level regressions.
- M006/S04: In this worktree, `apps/web/node_modules/.bin/playwright` exists but Playwright still tries to start `config.webServer` instead of reusing the ready port-3001 Next server; if verification fails immediately with `EADDRINUSE`, treat it as harness drift in `playwright.config.ts`/server reuse rather than a page-selector regression.
- M006/S05: If T01 reported env-contract drift but `packages/backend/scripts/verify-env-contract.ts` later passes, treat the drift as resolved state rather than reopening backend/domain code; verify the failure-path surface with `--check-failure-fixture`, then rerun the staged backend sweep and record the changed baseline explicitly.
- M006/S06: For native runtime proof, run `xcode-select -p` and `xcrun simctl list devices` before any Expo iOS launch; if the path is `/Library/Developer/CommandLineTools`, classify the machine as Xcode-path blocked, then still run `pnpm --filter native-app typecheck` and one `pnpm --filter native-app exec expo start --ios` attempt to prove whether the stop point stays outside app code.
- M006/S06: Keep native startup logging targeted. `apps/native/App.tsx` should suppress only known noisy warnings, not `LogBox.ignoreAllLogs()`, and `apps/native/ConvexClientProvider.tsx` should fail with sanitized missing-key names (`EXPO_PUBLIC_CONVEX_URL`, `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY`) so future simulator runs can separate env/provider boot failures from auth or screen rendering issues.
